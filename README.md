# TS HELPER v5.0

**TS HELPER** - это desktop-инструмент для техподдержки и внутренней IT-эксплуатации, который объединяет в одном окне работу с пользователями, инфраструктурой, телефонией и удалённой автоматизацией.

Проект задуман как **all-in-one рабочее место инженера поддержки**: быстро найти сотрудника, проверить его хост, открыть AD/GLPI, подключиться по SSH, пробросить ключ, поставить ПО и отследить входящий звонок.

---

## Идея проекта

Главная цель проекта - убрать рутину и разрозненность инструментов.

Вместо постоянного переключения между AD, GLPI, PowerShell, SSH, браузером, телефонией и локальными скриптами инженер получает **единый интерфейс**, который:

- хранит карточки пользователей,
- подтягивает данные из систем,
- даёт быстрые контекстные действия,
- автоматизирует типовые операции.

## Ключевой функционал

### 1. Единая база пользователей и удобный поиск

В приложении есть карточки пользователей с поиском по:

- ФИО,
- имени ПК,
- внутреннему номеру.

Данные можно:

- хранить локально,
- импортировать и экспортировать,
- вручную добавлять и редактировать.

Структура пользователя поддерживает не только основной ПК, но и альтернативные варианты хостов, что полезно в реальной эксплуатации, где имя машины может меняться или иметь несколько валидных вариантов.

---

### 2. Интеграция с Active Directory

TS HELPER умеет:

- забирать пользователей из AD,
- формировать записи на основе доменных данных,
- открывать карточку сотрудника в ADHelper напрямую из контекстного меню.

Для поддержки это означает более быстрый переход от поиска человека к реальным действиям по учётной записи.

---

### 3. Интеграция с GLPI

Проект подключается к GLPI через API, ищет пользователя, сопоставляет ему компьютер и умеет использовать GLPI как источник актуального имени хоста.

Это помогает:

- держать карточки в актуальном состоянии,
- автоматически корректировать основной ПК пользователя по инвентарным данным.

---

### 4. Телефония и CallWatcher

Одна из самых сильных возможностей проекта — встроенный **CallWatcher**.

Приложение:

- отслеживает звонки по заданным внутренним номерам,
- парсит страницу FreePBX,
- определяет, кто звонит,
- поднимает карточку пользователя в верх списка.

Дополнительно реализовано сопоставление:

- по внутреннему номеру,
- по ФИО,
- по мобильному номеру через AD.

Это особенно полезно для удалённых сотрудников.

---

### 5. Умная визуализация для оператора

Во время работы приложение показывает:

- статусы доступности,
- приоритет звонящих пользователей,
- всплывающие уведомления,
- быстрый доступ к действиям по карточке.

Отдельно проработана UI-производительность:

- дебаунс обновлений интерфейса,
- оптимизация перерисовки карточек,
- снижение нагрузки при потоке телефонных событий.

---

### 6. Удалённая работа по SSH

TS HELPER поддерживает SSH-сценарии как для интерактивного подключения, так и для automation.

Реализовано:

- подключение по логину/паролю,
- поддержка ключей,
- автопроброс публичного ключа,
- fallback-логика,
- учёт Windows-окружения,
- безопасная сборка команд для `sshpass`,
- поддержка `StrictHostKeyChecking`.

---

### 7. Проброс SSH-ключей и автоматическая подготовка доступа

Проект умеет:

- пробрасывать SSH-ключ на удалённый Linux-хост,
- при включённой опции сразу донастраивать managed sudoers-профиль для automation.

Это превращает ручную подготовку машины в понятный и повторяемый процесс из интерфейса.

---

### 8. Установка ПО на удалённые Ubuntu-хосты

В проект встроен модуль удалённой установки ПО.

Пользователь может:

- выбрать программу из каталога,
- управлять каталогом прямо из окна установки (добавление, редактирование, удаление или отключение),
- увидеть описание,
- включить принудительную переустановку,
- запустить установку в фоне.

Архитектура поддерживает разные сценарии:

- `apt`-пакеты,
- локальные `.deb`-файлы,
- `.deb` по URL,
- локальные скрипты,
- custom commands.

Для `apt` в каталоге поддерживается обратная совместимость со старым полем `package_name`, а также новый формат `packages`. Ввод в UI нормализуется до безопасного списка пакетов без shell-команд.

### 11. Windows Software Deployment (модуль v5.x)

В `v5.x` Windows deployment выделен в отдельный модуль и доведён до предсказуемого support-flow без смешивания с Linux-логикой.

Что поддерживается сейчас:

- отдельный каталог `software_catalog_windows.json` с валидацией, атомарным сохранением и ротацией backup-файлов;
- backend-aware detection: проверки выполняются в том же контексте, что и установка (`local_subprocess` локально, `psexec` на целевом хосте);
- pre/post detection с типами `file_exists`, `registry_exists`, `registry_value`, `uninstall_display_name`, `product_code`, `command_success`, `powershell_script`;
- для `registry_value` реализованы операторы `==`, `!=`, `>`, `<`, `>=`, `<=` с диагностикой текущего и ожидаемого значения;
- backend-абстракция включает подготовку payload, detection, install и cleanup;
- для `psexec` используется явный staging в admin-share (`\HOST\C$...`) без неявного fallback на localhost;
- UI-диалог показывает актуальный backend/target/mode и блокирует install/check, если для `psexec` не задан target host.

Ограничения первой версии (честно):

- `msix`, `winget` и `powershell_script` в remote+SYSTEM сценариях зависят от окружения целевого хоста и прав, поэтому требуют предварительной проверки в вашей инфраструктуре;
- записи каталога считаются доверенными (trusted admin catalog), поэтому любые custom command/script должны проходить внутренний review;
- модуль не делает «магический» fallback при ошибках transport/доступа, а возвращает диагностический статус (`invalid_target`, `transport_failed`, `source_unavailable`, `detection_failed`, `elevation_required`, `timeout`).

Новые/обновлённые настройки (Settings → SSH):

- путь к Windows-каталогу;
- backend по умолчанию (`local_subprocess` / `psexec`);
- путь к `PsExec64.exe`;
- timeout по умолчанию;
- флаг пропуска установки, если пакет уже обнаружен;
- флаг предпочтения SYSTEM-контекста;
- remote temp directory.

### 9. Работа с интерактивными установщиками

Если установщик на удалённой машине требует ответ оператора, TS HELPER не зависает и не ломает процесс.

Вместо этого он:

- показывает отдельный диалог,
- выводит prompt,
- ждёт решение пользователя.

Это важная практическая доработка, потому что в реальной поддержке не все сценарии установки полностью non-interactive.

---

### 10. Автоматическое исправление sudo-доступа

Если во время удалённой установки выясняется, что на хосте не настроен passwordless sudo, проект умеет корректно это обрабатывать:

- выявить проблему,
- запросить подтверждение,
- запросить sudo-пароль,
- создать и проверить managed sudoers-файл,
- продолжить установку.

## Возможности
- Управление списком пользователей (добавление, редактирование, импорт/экспорт).
- Синхронизация с Active Directory и проверка доступности хостов.
- Синхронизация с GLPI для подбора префиксов ПК и выбор основного устройства при наличии нескольких вариантов.
- Запуск SSH/административных команд с учётом сохранённых настроек.
- Remote automation для Ubuntu-хостов по SSH: проброс ключа и установка ПО из JSON-каталога без изменения интерактивного SSH-сценария.
- Мониторинг звонков (Call Watcher) с парсингом страниц FreePBX.
- Логирование действий в `app.log`; дампы PBX в `_pbx_debug` доступны только при включённом debug-флаге в настройках телефонии.
- Сворачивание приложения в системный трей или прозрачный мини-виджет поверх всех окон.

## Требования
- Python 3.9+ (разработка ориентирована на Windows, хранение паролей завязано на Windows-хранилища).
- Виртуальная среда (рекомендуется) и зависимости из `requirements.txt`.
- Для remote automation по SSH установите `paramiko` (`pip install paramiko`).
- Наличие `tkinter` в установленной сборке Python.
- Для функций шифрования DPAPI и интеграции с AD требуется Windows.
- Для безопасного хранения паролей используется библиотека `keyring` и доступное в системе защищённое хранилище учётных данных (Windows Credential Manager). При отсутствии `keyring` используется резервное DPAPI-хранилище в файле `.ts_help_ad_secrets.json` рядом с конфигурацией.

## Установка и запуск
```bash
python -m venv .venv
source .venv/bin/activate  # или .venv\\Scripts\\activate в Windows
pip install -r requirements.txt
python "TS HELPER V5.0.py"
```

## Настройка
Основные параметры задаются в `config.json`:
- `ad_username` — учётные данные для подключения к AD (пароль хранится в keyring по ссылке из конфигурации).
- `cw_url`, `cw_cookie`, `cw_exts` — доступ к странице статистики FreePBX и отслеживаемые внутренние номера.
- `cw_ui_debounce_ms` — задержка батчинга UI-обновлений CallWatcher (по умолчанию 150 мс), помогает избежать подтормаживаний интерфейса при частых событиях звонков.
- `pbx_debug_dump` — включение записи технических дампов FreePBX (`_pbx_debug`). По умолчанию отключено, чтобы не создавать лишнюю дисковую нагрузку в рабочем режиме.
- `cw_login` — учётные данные FreePBX (пароль хранится в keyring).
- `ssh_login`, `ssh_terminal` — запуск удалённых сессий (пароль хранится в keyring).
- `ssh_key_enabled`, `ssh_key_private_path`, `ssh_key_public_path`, `ssh_key_auto_bootstrap`, `ssh_key_push_nopasswd` — отдельные настройки automation-слоя для Ubuntu по SSH-ключу. Опция `ssh_key_push_nopasswd` позволяет после успешного проброса ключа по отдельному подтверждению создать managed-файл `/etc/sudoers.d/90-support-tshelper` для пользователя из `ssh_login`; перед применением выполняются проверка логина, проверка `sudo -n true`, попытка использовать `SSH Password` как sudo-пароль и безопасная валидация через `visudo -cf`.
- `software_catalog_path` — путь к JSON-каталогу ПО для окна «Установка ПО». Если файла нет, приложение создаст шаблон автоматически.
- `glpi_api_url`, `glpi_app_token`, `glpi_user_token`, `glpi_prefix_field` — доступ к GLPI API (apirest.php) для получения закреплённых ПК и префиксов. Токены сохраняются в защищённом хранилище. Поле `glpi_prefix_field` задаёт имя поля в ответе GLPI, откуда брать префикс/тип ОС (например, `name` или пользовательское поле). В списке устройств игнорируются хосты с префиксами `wr-` и `lr-`.
- `glpi_verify_ssl` — проверять SSL-сертификат при обращении к GLPI. Для самоподписанных сертификатов снимите галочку в настройках или укажите путь к доверенному сертификату через системное хранилище.
- `glpi_use_in_ad_sync` — использовать ли сверку с GLPI при синхронизации AD. Отключение помогает временно пропускать обращения к GLPI, если там есть проблемы с сертификатами или доступностью.
- Геометрия окон (`window_geometry` и др.) сохраняется автоматически при закрытии.

Пароли (`ad_password`, `ssh_password`, `cw_password`, `reset_password`) сохраняются только в защищённом хранилище: в Windows Credential Manager через `keyring` либо в резервном DPAPI-файле рядом с конфигурацией. В `config.json` остаются только идентификаторы записей. При недоступности защищённого хранилища пароли не отображаются в настройках, выводится предупреждение и значения сохраняются лишь временно в памяти.

Список отображаемых пользователей хранится в `users.json`. Конфигурация док-панели (имена кнопок и их пути/URL) сохраняется в отдельном файле `dock_items.json` рядом с конфигурацией, чтобы проще переносить окружение между машинами.

## Известные особенности
- Обновления с GitHub проверяются в фоне; при отсутствии `requests` проверка пропускается.
- Для ADHelper.py добавлен мастер «Увольнение» (только домен omg-cspfmba). Для безопасной проверки без изменений AD установите `ADHELPER_DRYRUN=1`: шаги Set/Disable/Move будут симулироваться и логироваться как тестовые.
- Для ADHelper.py поддержана генерация одного приветственного ODT после успешного создания пользователя (без разбиения по доменам). Шаблон берётся из файла `New User.odt` рядом со скриптом. Предпочтительный режим — плейсхолдеры `{{LOGIN}}`, `{{EMAIL}}`, `{{PASSWORD}}` в тексте документа; fallback-режим умеет заменять значения в строках с префиксами `Логин:`, `Ваш Email:`, `Временный пароль:`.
- Для полей `{{LOGIN}}`, `{{DOMAIN_LOGIN}}`, `{{EMAIL}}`, `{{PASSWORD}}` в `New User.odt` используйте обычный пропорциональный шрифт (например, Calibri/Arial/Carlito/Liberation Sans) без моно-стиля, разрежения символов и нестандартного kerning: генератор заменяет только текст плейсхолдеров и не назначает шрифт в коде.
- Сгенерированные приветственные файлы сохраняются в `%APPDATA%\ADHelper\generated` с уникальным именем и могут быть отправлены на печать на принтер по умолчанию только после подтверждения оператора.
- Логика DPAPI (`pywin32`) работает только в Windows; на других ОС защищённое хранилище может быть недоступно, поэтому пароли не сохраняются и не отображаются.
- При использовании Call Watcher убедитесь, что cookie FreePBX актуальна, иначе парсинг будет отображать страницу входа.
- Логи фиксируют источник выбранного префикса (AD/GLPI), а также заменённые логины и переключения основного ПК.
- При отсутствии `paramiko` приложение продолжает запускаться; ошибка появится только при использовании remote automation-функций.
